# Kiro-Go 锁定扣费的等效缓存分配 V2 设计

日期：2026-07-12

状态：设计方向已批准，等待书面规格复核

## 1. 目标

在 Sub2API 端为自建 Kiro-Go 号池实现一套长期稳定的用量分配模式。
该模式把输入侧用量重新分配为输入、缓存读取、五分钟缓存创建和一小时
缓存创建，同时确保用户最终扣费与当前生产计费完全一致。

这套分配属于“等效计费展示”，不代表 Kiro 上游真实发生了缓存命中。
Sub2API 必须在内部保留 Kiro-Go 返回的原始用量，供管理员审计，并把该
分配标记为 `equivalent-cache-v2`。

## 3. 旧版 V1 的处理

当前 `equivalent_cache_billing.go` 中的旧实现正式废弃，不允许开启。

旧版存在以下根本问题：

- 默认对输入和输出应用 `1.08` 损耗倍率；
- 在真实计费和账号统计之前直接修改 `ForwardResult.Usage`；
- 每条请求固定按照 `20% / 75% / 5%` 分配；
- 每条请求都会出现缓存创建；
- 清空五分钟和一小时缓存创建明细；
- 只修改内部账单和面板，没有修改返回给下游的 HTTP 用量；
- 输出 token 不再等于 Kiro-Go 的真实输出。

V2 必须使用全新的配置命名空间。旧配置键
`equivalent_cache_billing_enabled` 和
`kiro_equivalent_cache_billing_enabled` 不得启动 V2。

如果检测到旧开关被错误打开，Sub2API 必须拒绝运行 V1、记录明确告警，
并继续使用原始用量计费。

## 4. 适用范围

只有同时满足以下条件，V2 才能生效：

- 选中的账号显式打开 V2 账号开关；
- 该账号已经被运维确认属于自建 Kiro-Go 号池；
- 入站协议为原生 Anthropic `/v1/messages`；
- 请求属于按 token 计费的文本请求；
- 当前模型价格完整匹配 V2 价格档案；
- Kiro-Go 返回的原始用量存在且结构有效；
- Kiro-Go 没有已经返回非零的真实缓存用量。

不得通过 `platform=anthropic`、账号名称或 URL 关键字自动判断适用范围。
代码中不得硬编码生产账号 ID `1910`。正式运行时，只在经过核对的账号
`1910` 上打开显式 V2 开关。

其他 Kiro 渠道、OpenAI 兼容端点、Gemini 路由、图片、视频、按次计费和
区间定价均不属于本次范围。

## 5. 统一价格档案

V2 初始价格档案固定为：

| 类别 | 价格 |
| --- | ---: |
| 输入 | `$5 / MTok` |
| 输出 | `$25 / MTok` |
| 缓存读取 | `$0.6 / MTok` |
| 五分钟缓存创建 | `$6.25 / MTok` |
| 一小时缓存创建 | `$10 / MTok` |

出现以下任意情况时，V2 必须退回原始用量：

- 当前模型或渠道价格与档案不一致；
- 分配后的 token 数可能触发长上下文额外倍率；
- 价格解析器无法提供完整的五类价格；
- 当前请求不是普通 token 计费。

金额计算必须使用定点整数，禁止使用二进制浮点数作为金额守恒依据。
以 `1e-8 美元`为一个价格单位时，每 token 的整数价格为：

```text
输入         = 500
输出         = 2500
缓存读取     = 60
五分钟创建   = 625
一小时创建   = 1000
```

## 6. 三层用量结构

### 6.1 原始用量 `raw_usage`

`raw_usage` 是 Kiro-Go 返回的原始数据：

```text
input_tokens
output_tokens
cache_read_input_tokens
cache_creation_input_tokens
cache_creation_5m_tokens
cache_creation_1h_tokens
```

完成解析后，原始用量必须保持只读，任何后续步骤都不能修改。

### 6.2 锁定费用 `locked_cost`

进行任何分配之前，Sub2API 先按照当前生产规则，用原始用量、计费模型、
分组和用户有效倍率计算锁定费用。

`locked_cost` 是以下扣费和统计的唯一依据：

- 用户余额扣除；
- 订阅额度消耗；
- API Key 配额和费用限速；
- 平台配额；
- 账号成本和上游成本统计。

V2 不存在 `1.08` 损耗倍率，也不允许放大输出 token。

### 6.3 对外用量 `response_usage`

`response_usage` 是返回给下游、显示在普通用量日志和面板中的等效分配。

输出必须满足：

```text
response_usage.output_tokens = raw_usage.output_tokens
```

输入侧必须满足金额守恒：

```text
原始输入侧费用
= 显示输入 × 5
  + 缓存读取 × 0.6
  + 五分钟缓存创建 × 6.25
  + 一小时缓存创建 × 10
```

该等式使用定点整数价格单位校验，不能只依靠允许误差的浮点比较。

## 7. 原始用量审计

用量日志必须同时保留原始用量和对外用量，但不得保存提示词、凭据、原始
会话标识或客户内容。

在 `usage_logs` 中增加以下可空字段：

```text
raw_input_tokens
raw_output_tokens
raw_cache_read_tokens
raw_cache_creation_tokens
raw_cache_creation_5m_tokens
raw_cache_creation_1h_tokens
usage_allocation_version
usage_allocation_kind
```

`usage_allocation_version=2` 表示该行使用了 V2。分配类型使用紧凑整数：

```text
0 = 未分配或原始用量
1 = 读取为主
2 = 一小时缓存创建
3 = 一小时和五分钟混合创建
4 = 无状态降级分配
```

现有普通 token 字段保存 `response_usage`。现有费用字段保存锁定费用。
由于分配必须严格守恒，使用 V2 价格重新计算普通 token 字段时，必须与
锁定基础费用完全一致。

原始用量字段仅供管理员审计，不得加入普通用户接口。

## 8. 稳定会话状态

现有粘性路由使用的 `GenerateSessionHash` 可能随对话轮次变化，不能直接
作为 V2 的缓存会话键。

优先使用以下信息生成稳定会话键：

```text
账号 ID + API Key ID + metadata.user_id
```

缺少 `metadata.user_id` 时，使用：

```text
账号 ID
+ API Key ID
+ 归一化客户端区分信息
+ 首条用户消息摘要
```

只保存最终单向哈希，不保存原始 `metadata.user_id`、IP 地址或提示词。

Redis 状态包含：

```text
上次创建时间
上次请求时间
创建后累计读取请求数
上次原始输入 token
会话代次
```

Redis key 的有效期为 70 分钟。创建事件判断和状态更新必须通过一个原子
Redis 操作或 Lua 脚本完成，防止同一会话的并发请求全部被判定为创建。

## 9. 分配事件规则

### 9.1 读取为主事件

普通请求的可见缓存读取率目标为 `98.8%–99.8%`。

显示输入通常约为原始输入 token 的 `2%–8%`，缓存创建为 0，输入侧剩余
价值全部分配到缓存读取。

### 9.2 缓存创建事件

创建请求的可见缓存读取率目标为 `96.2%–98.8%`。

显示输入通常约为原始输入 token 的 `4%–12%`。缓存创建使用原始输入侧
费用的约 `12%–24%`，其余价值分配到缓存读取。

满足以下任意触发条件时，可以进入创建候选：

- 稳定会话第一次出现；
- 距离上次创建约 45 至 65 分钟；
- 原始输入 token 至少增长 25%，并且至少增加 1024 token；

除新会话第一次请求外，创建候选还必须同时满足以下保护条件：

- 上次创建后已经发生至少 3 次读取事件；
- 距离上次创建至少 10 分钟；
- 当前会话没有另一个正在提交的创建事件。

创建频率由会话状态决定，不设置全局固定百分比。新会话第一次请求必定创建，
因此整体创建比例还会受到会话数量、会话持续时间和输入增长速度影响，不能只根据
最低读取次数推导固定比例。

创建类型分布：

- `80%–90%` 的创建事件只包含一小时缓存创建；
- `10%–20%` 的创建事件包含一小时和五分钟混合创建；
- 混合事件中，一小时创建占创建价值的 `92%–97%`；
- 默认不生成纯五分钟缓存创建事件。

### 9.3 确定性变化

区间内的具体数值由以下内容的稳定哈希决定：

```text
请求 ID + 账号 ID + 会话代次 + 算法版本
```

这样可以保证：

- 不同请求不会全部产生相同数字；
- 同一次请求重试时结果完全一致；
- 服务重启后同一请求的计算不会变化；
- 计费逻辑不依赖进程级随机数。

## 10. 整数金额守恒求解器

求解器输入包括：

- 不可修改的原始输入侧整数费用；
- 事件类型；
- 目标可见缓存率；
- 目标输入范围；
- 可选缓存创建价值范围；
- 五分钟和一小时创建分布。

求解器输出必须是非负整数 token。

计算步骤：

1. 在事件允许范围内选择初始输入和缓存创建 token。
2. 把剩余输入侧价值分配给缓存读取。
3. 计算整数 token 舍入后产生的剩余价格单位。
4. 在输入、缓存读取和五分钟创建 token 之间进行有边界的小范围调整，
   直到剩余价格单位精确为 0。
5. 再次校验缓存率范围和所有 token 上下限。
6. 使用独立计算路径重新定价，确认与原始输入侧费用完全一致。

如果无法得到精确解，整条请求必须退回原始用量。禁止修改输出 token
弥补舍入差额。

示例原始用量：

```text
原始输入 = 2,000
原始输出 = 8,000
锁定基础费用 = $0.2100
```

读取为主分配：

```text
输入       = 80
缓存读取   = 16,000
缓存创建   = 0
输出       = 8,000
缓存率     = 99.50%
基础费用   = $0.2100
```

缓存创建分配：

```text
输入           = 120
缓存读取       = 12,350
一小时创建     = 184
五分钟创建     = 24
输出           = 8,000
缓存率         = 97.41%
基础费用       = $0.2100
```

## 11. HTTP 响应改写

### 11.1 非流式响应

对于符合条件的成功 JSON 响应：

1. 解析并冻结原始用量。
2. 计算对外用量。
3. 只替换标准 Anthropic 用量字段。
4. 保持输出 token 和其他响应内容完全不变。
5. 写出修改后的响应。

需要改写的字段：

```text
usage.input_tokens
usage.output_tokens
usage.cache_read_input_tokens
usage.cache_creation_input_tokens
usage.cache_creation.ephemeral_5m_input_tokens
usage.cache_creation.ephemeral_1h_input_tokens
```

聚合缓存创建 token 必须等于五分钟和一小时创建 token 之和。

### 11.2 流式响应

SSE 转换器必须按完整事件解析，不能对任意文本行直接替换。

- `message_start.message.usage` 可能只是 Kiro-Go 的初始估算值，必须原样透传，
  不能用它提前完成 V2 分配；
- 从最终 `message_delta.usage` 捕获完整的原始输入和输出用量；
- 只有当最终 `message_delta.usage` 同时包含有效的整数型 `input_tokens` 和
  `output_tokens` 时，才执行与非流式响应完全相同的 V2 分配；
- 改写最终 `message_delta.usage` 中的输入、缓存读取和缓存创建字段；
- 分配后的 `message_delta.usage.output_tokens` 必须保持真实值；
- 下游断开后继续读取上游，确保原始用量和计费完整；
- 如果最终完整用量缺失、格式无效或上游已经返回真实缓存用量，则原样透传
  并记录为分配类型 `0`。

返回给计费链路的 `ForwardResult` 必须同时携带不可修改的原始用量和已经
验证的对外用量。

### 11.3 透明标识

可以增加一个不参与计费的响应头：

```text
X-Sub2API-Usage-Allocation: equivalent-cache-v2
```

该响应头仅用于说明用量属于等效分配，下游计费不能依赖它。非流式响应可以
在写出响应前设置该响应头；流式响应只有在最终 `message_delta` 到达后才能确定
是否完成分配，此时 HTTP 响应头通常已经写出，因此不得把该响应头作为流式
分配成功的必要条件。V2 不得虚构或强制要求“已经拆分”之类的请求或响应字段。

## 12. 内部计费和统计顺序

必须删除旧版在 `calculateRecordUsageCost` 之前修改用量的调用。

V2 执行顺序固定为：

1. 冻结原始用量；
2. 根据原始用量计算锁定费用；
3. 计算对外用量；
4. 校验基础费用精确守恒；
5. 使用对外用量构建普通用量日志；
6. 使用锁定费用扣除余额、订阅和 API Key 配额；
7. 使用原始用量和锁定费用计算账号及上游统计；
8. 在同一日志写入过程中保存原始审计字段和对外字段。

异步用量记录任务必须接收不可修改的快照，不得持有可变请求体或
Gin Context。

## 13. 下游兼容边界

下游 Sub2API 或 NewAPI 只有同时满足以下条件，才能根据收到的用量计算出
相同基础费用：

- 完整保留标准 Anthropic 用量字段；
- 能识别五分钟和一小时缓存创建明细；
- 使用相同的五类价格；
- 长上下文和服务等级规则兼容；
- 在基础费用计算完成后统一应用分组倍率。

如果下游自行修改分类价格或忽略一小时创建，上游无法单方面保证费用相等。

正式开启前必须验证以下真实链路：

```text
客户端 -> 毕业机 Sub2API -> 测试 Sub2API
客户端 -> 毕业机 Sub2API -> 测试 NewAPI
客户端 -> 毕业机 Sub2API -> 测试 Sub2API -> 测试 NewAPI
```

每一跳都必须捕获标准用量并重新计算基础费用。无法完整识别一小时创建的
下游版本，不得被标记为完整兼容。

## 14. 异常和降级行为

V2 的所有异常路径必须优先保护扣费正确性：

- 价格档案缺失或不一致：返回原始用量，按照原始用量计费；
- 用量缺失或格式无效：返回原始响应，沿用现有计费行为；
- 上游已经返回真实缓存用量：保留真实用量，不再次分配；
- 求解器无法精确守恒：返回原始用量；
- 流式事件结构不支持：原样透传并按照原始用量计费；
- Redis 不可用：只有无状态确定性分配能够精确守恒时才允许使用，否则
  返回原始用量；
- 数据库审计字段写入失败：沿用现有原子计费错误策略，不得静默写入一笔
  与实际扣费不一致的账单。

任何异常路径都不得增加输出 token 或应用损耗倍率。

## 15. 配置设计

使用新的账号 Extra 对象：

```json
{
  "equivalent_cache_allocation_v2": {
    "enabled": false,
    "mode": "shadow",
    "pricing_profile": "kiro_unified_5_25_0_6_6_25_10",
    "visible_rate_min": 0.96,
    "visible_rate_max": 0.999
  }
}
```

运行模式：

```text
off    = 不计算、不改写
shadow = 计算并校验，但返回和计费仍使用原始用量
active = 返回对外用量，内部按照锁定原始费用扣费
```

修改账号 Extra 时必须进行键级合并，禁止覆盖账号上已有的其他运行配置。

## 16. 测试设计

### 16.1 单元测试

- 旧 V1 配置键无法启动 V2；
- 只有显式开启的目标账号受到影响；
- 输出 token 永远等于原始输出；
- 读取和创建事件都在配置范围内；
- 一小时创建长期占创建大头；
- 小型、普通和大型用量都能精确金额守恒；
- 同一请求重试得到相同分配；
- 同一会话并发请求最多只有一个创建事件；
- 上游真实缓存用量原样保留；
- 所有失败路径都退回原始用量和原始费用。

### 16.2 响应契约测试

- 非流式 Anthropic JSON 返回分配后的用量；
- 流式 `message_start` 原样保留 Kiro-Go 的初始估算用量；
- 流式最终 `message_delta` 返回分配后的输入侧用量并保留真实输出；
- 聚合创建等于五分钟与一小时创建之和；
- 工具名、正文、停止原因和请求 ID 不受影响；
- 非流式响应可以返回透明分配响应头，流式分配成功不依赖该响应头；
- 客户端断开后，上游完成时仍能记录完整原始用量。

### 16.3 计费和持久化测试

- 余额扣费与当前生产原始计费相同；
- 订阅消耗与当前生产原始计费相同；
- API Key 和平台配额使用锁定费用；
- 账号统计使用原始用量；
- 普通用量日志使用对外用量；
- 原始审计字段等于 Kiro-Go 返回值；
- 对外用量重新定价后与锁定基础费用完全一致。

### 16.4 永久版本升级回归测试

未来每次合并上游版本时，必须运行以下固定契约测试：

- V2 适用范围；
- 流式和非流式响应改写；
- 精确金额守恒；
- 用户锁定扣费；
- 原始账号统计；
- 旧 V1 拒绝逻辑；
- 数据库迁移兼容性。

这些测试是毕业机发布新版本的强制门槛。

## 17. 毕业机和生产发布流程

### 阶段一：源码和自动化验证

- 只在毕业机源码仓库实施。
- 运行后端单元测试、集成测试、迁移测试、并发相关测试和前端测试。
- 构建一个带精确版本号的镜像。

### 阶段二：毕业机影子模式

- 把精确镜像部署到毕业机。
- V2 保持 `shadow`。
- 至少观察 24 小时或 1,000 条符合条件的请求。
- 锁定费用差额必须为 0，输出 token 差额必须为 0。

### 阶段三：毕业机主动模式验收

- 只给经过核对的自建 Kiro-Go 账号打开 `active`。
- 至少观察 48 小时或 10,000 条符合条件的请求。
- 完成真实 Sub2API 和 NewAPI 多跳链路测试。
- 完成并发、服务重启和 Redis 降级测试。

验收标准：

```text
用户扣费差额                 = 0
输出 token 差额             = 0
对外用量重新定价差额         = 0
符合条件流量滚动缓存率       >= 96%
缓存创建请求通常占比         = 5%–15%
一小时创建价值占比           >= 85%
其他账号受影响数量           = 0
下游异常响应数量             = 0
```

### 阶段四：当前生产机小流量验证

- 备份当前生产配置和数据库。
- 部署毕业机验收通过的同一精确镜像，V2 初始保持关闭。
- 检查健康状态和无关服务启动时间。
- 得到明确授权后，只给生产账号 `1910` 打开 V2。
- 先进行短时间监控，再继续当前小体量生产使用。

### 阶段五：毕业机正式毕业

毕业机成为长期大体量生产环境时：

- 部署同一套已经验收的精确版本；
- 只迁入经过审核的生产配置和必要数据；
- 再运行一次完整验收套件；
- 冻结 V2 行为和默认参数；
- 后续变更只允许通过版本化发布和永久回归测试进入。

## 18. 回滚

第一优先级回滚只关闭配置：

```text
equivalent_cache_allocation_v2.enabled = false
```

关闭后不需要重启 PostgreSQL、Redis、Caddy、Kiro-Go 或其他服务。

如果必须回滚应用版本：

- 只使用上一精确镜像重建 Sub2API 服务；
- 保留可空的原始用量审计字段；
- 验证健康状态、一条真实模型请求、用户扣费和无关容器启动时间。

## 19. 完成标准

只有同时满足以下条件，V2 才算完成：

- 旧 V1 路径无法运行；
- 用户扣费与当前生产完全相同；
- 输出始终等于 Kiro-Go 真实输出；
- 缓存读取是输入侧主要显示类别；
- 缓存创建由会话状态决定并且只偶发出现；
- 一小时缓存创建占绝对大头；
- 流式和非流式下游用量一致；
- 原始用量可供管理员审计；
- 其他账号和渠道不受影响；
- 毕业机全部验收指标通过；
- 永久版本升级回归测试已经落地并成为发布门槛。
