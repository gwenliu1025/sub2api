# Equivalent Cache V2 账号级资格、动态价格与 Kiro-Go 内网修复设计

日期：2026-07-14

状态：设计已确认，等待书面规格复核

## 1. 背景

生产账号 `1910 / goep-kiro-power` 已显式开启 Equivalent Cache V2，但该账号从
分组 `101` 切换到分组 `84` 后，V2 记录立即消失：

- 最后一条 V2 记录时间：`2026-07-13 22:59:08 +08`
- 第一条分组 `84` 记录时间：`2026-07-13 23:00:12 +08`
- 核验时分组 `84` 共 510 条记录，V2 记录为 0
- 其中 Opus 4.6 共 209 条，缓存读取和缓存创建均为 0

账号上的 V2 配置仍然存在且处于 `active`。失效原因不是线程堵塞，也不是账号
开关丢失，而是代码把当前分组价格是否严格匹配固定
`$5/$25/$0.6/$6.25/$10` 档案作为第二道启用门槛：

- 分组 `101` 的缓存读取价格为 `$0.6/MTok`
- 分组 `84` 的缓存读取价格为 `$0.5/MTok`
- 价格不完全相同时，响应阶段和计费阶段都会静默退回原始用量

生产 Kiro-Go 还存在第二个独立问题：

- Sub2API 位于 Docker 网络 `sub2api_sub2api-network`
- Kiro-Go 位于 Docker 网络 `kiro-go-pr131_default`
- 两个容器没有共享网络
- 账号 Base URL 当前使用 `http://154.36.172.65:8321`
- 请求从 Sub2API 容器绕到宿主机公网发布端口，没有使用 Docker 内网直连

本设计修复上述两个根因。它覆盖
`2026-07-12-kiro-go-cost-locked-equivalent-cache-v2-design.md` 中要求固定价格档案
匹配才能执行 V2 的条款；其他已经实现的 V2 行为继续有效。

## 2. 目标

- V2 的业务启用资格只由上游账号的 Extra 配置决定。
- 账号迁移分组时，不能因为分组编号或固定价格档案不同而关闭 V2。
- 每条请求使用其实际生效的 token 价格执行整数金额守恒。
- 响应改写和异步计费使用同一份不可变价格快照。
- 保持用户扣费、订阅消耗、API Key 配额和账号成本与原始用量计费一致。
- 保持流式与非流式请求使用相同价格和相同分配算法。
- Sub2API 通过 Docker 共享网络直连 Kiro-Go。
- Kiro-Go 宿主机端口不再暴露到公网。
- README、中文发布说明和运维状态文档反映新的账号级语义与内网拓扑。
- 继续重建 `0.1.152`，不得发布 `0.1.153`。

## 3. 不在本次范围内

- 不修改当前缓存读取、缓存创建和稳定提升比例。
- 不修改“三次读取”和“十分钟保护”规则。
- 不修改 V2 会话键和 Redis 状态结构。
- 不让上游已经返回真实缓存用量的请求再次分配。
- 不支持按次计费、区间价格或长上下文动态倍率。
- 不修改 PostgreSQL、Redis、Caddy、CPA、QQ/NapCat 或其他无关服务。
- 不改变 Kiro-Go 的业务逻辑、账号池内容或认证数据。

## 4. 方案比较

### 4.1 方案 A：账号资格与请求价格分离

账号 Extra 决定 V2 是否启用；分组和模型解析器只提供本次请求的实际价格。
价格被转换为整数价格向量，并随请求一路传递到响应和计费阶段。

优点：

- 账号换分组不会丢失 V2 资格。
- 不覆盖分组原有价格，现有计费语义保持不变。
- `$0.5`、`$0.6` 及其他可表示的平坦 token 价格都能使用。
- 响应和计费不会因为二次解析价格而发生竞态或不一致。

缺点：

- 分配器和计费校验需要从硬编码常量改为显式价格向量。
- 需要增加一组跨阶段快照字段。

### 4.2 方案 B：账号固定价格覆盖分组价格

在账号配置中保存固定价格，V2 和计费都忽略当前分组价格。

该方案会让账号价格覆盖用户所属分组的计费配置，容易造成面板价格、用户扣费和
渠道价格语义分裂，因此不采用。

### 4.3 方案 C：为每个分组维护固定价格档案

为 `$0.5`、`$0.6` 等价格分别增加档案，并继续要求分组价格命中某个档案。

该方案仍然让分组控制账号 V2 资格。每次新增价格或迁移分组都可能再次静默失效，
因此不采用。

### 4.4 决策

采用方案 A：账号决定资格，实际价格决定计算，Kiro-Go 使用 Docker 内网。

## 5. 账号级启用资格

V2 业务资格只检查账号 Extra 中的以下内容：

```json
{
  "equivalent_cache_allocation_v2": {
    "enabled": true,
    "mode": "active",
    "visible_rate_min": 0.96,
    "visible_rate_max": 0.999,
    "kiro_go_pool_confirmed": true
  }
}
```

资格条件为：

- `enabled=true`
- `mode` 为 `shadow` 或 `active`
- `kiro_go_pool_confirmed=true`
- 可见缓存率上下界有效
- 旧 V1 开关没有被错误打开

现有 `pricing_profile` 字段继续允许保留，保证生产账号 Extra 无需立即迁移，但它
不再参与 V2 启用判断，也不再决定价格。新增账号可以不写该字段。

以下信息不得作为业务资格：

- 分组 ID
- 渠道 ID
- 账号名称
- Base URL 中的关键字
- 固定模型价格档案

分组仍可作为当前请求价格的来源。没有分组、价格缺失或价格无法安全表示时，仅
当前请求退回原始用量；这属于金额守恒保护，不代表账号 V2 被关闭。

## 6. 动态价格向量

### 6.1 数据结构

新增不可变的 V2 价格向量，使用 `1e-8 美元/token` 作为整数单位：

```text
input_unit
output_unit
cache_read_unit
cache_creation_5m_unit
cache_creation_1h_unit
```

例如：

```text
$5/MTok     -> 500
$25/MTok    -> 2500
$0.5/MTok   -> 50
$0.6/MTok   -> 60
$6.25/MTok  -> 625
$10/MTok    -> 1000
```

所有价格必须为正数，并且能够无损转换为整数价格单位。二进制浮点数不能直接作为
金额守恒依据。

### 6.2 请求价格解析

响应计划创建时，使用该请求最终确定的计费模型和当前实际分组调用现有价格解析器。
技术支持范围保持为：

- token 计费模式
- 完整支持缓存读取、五分钟创建和一小时创建价格
- 无区间价格
- 无长上下文输入或输出倍率
- 五类价格都能转换为正整数价格单位

不满足这些条件时只降级当前请求，不修改账号配置，也不缓存“账号不合格”的状态。

### 6.3 单次解析与快照传递

价格只能在响应计划创建时解析一次。生成的价格向量必须依次写入：

```text
equivalentCacheV2ResponsePlan
-> EquivalentCacheAllocation
-> equivalentCacheV2ResponseResult / streamingResult
-> ForwardResult
-> prepareEquivalentCacheV2BillingResults
```

异步计费阶段不得再次根据 API Key 分组调用价格解析器来判断 V2 是否有效。这样可
避免以下竞态：

- 请求响应后、异步计费前账号被迁移到其他分组
- 价格配置在两个阶段之间被刷新
- 响应阶段和计费阶段使用不同模型映射结果

价格向量只在进程内传递，不写入普通用户可见响应，不新增数据库迁移。

## 7. 金额守恒

分配器不再引用固定价格常量，而是显式接收当前请求的价格向量。

原始输入侧费用为：

```text
raw_input_cost =
  raw_input_tokens * input_unit
  + raw_cache_read_tokens * cache_read_unit
  + raw_cache_creation_5m_tokens * cache_creation_5m_unit
  + raw_cache_creation_1h_tokens * cache_creation_1h_unit
```

当前 V2 只处理上游真实缓存用量全为 0 的请求，因此正常情况下原始费用等于
`raw_input_tokens * input_unit`。

对外输入侧费用为：

```text
response_input_cost =
  response_input_tokens * input_unit
  + response_cache_read_tokens * cache_read_unit
  + response_cache_creation_5m_tokens * cache_creation_5m_unit
  + response_cache_creation_1h_tokens * cache_creation_1h_unit
```

成功分配必须满足：

```text
raw_input_cost = response_input_cost
response_output_tokens = raw_output_tokens
```

`EquivalentCacheAllocation.Valid` 必须使用分配时携带的同一价格向量重新校验。
计费阶段还要独立校验一次该向量、原始用量和对外用量。任何字段缺失、非正、
溢出或不守恒时，整条请求退回原始用量和原始计费。

输出价格随快照传递，用于确认本次请求价格完整并为后续审计保留一致语义；V2 不
改变输出 token，因此输入侧求解器不使用输出价格补偿差额。

## 8. 响应与计费数据流

### 8.1 非流式请求

1. 根据账号 Extra 判断 V2 资格。
2. 使用最终计费模型和当前实际分组解析价格向量。
3. 冻结 Kiro-Go 原始用量。
4. 使用价格向量执行 V2 分配和整数守恒校验。
5. 成功时改写标准 Anthropic usage 字段。
6. 将原始用量、对外用量、分配版本、分配类型和价格向量写入 `ForwardResult`。
7. 异步计费直接使用 `ForwardResult` 中的快照复核，不重新解析分组价格。

### 8.2 流式请求

- `message_start` 继续原样透传，不使用初始估算用量执行 V2。
- 最终 `message_delta.usage` 到达后，使用响应计划中的价格向量执行 V2。
- 流式结果必须携带与非流式相同的原始用量、对外用量、版本、类型和价格向量。
- 客户端提前断开后仍继续排空上游并完成相同的计费快照。

### 8.3 影子模式

`shadow` 模式使用动态价格向量计算和校验，但返回、日志计费和扣费仍使用原始
用量。影子日志继续只记录结果类型和守恒状态，不记录提示词、密钥或敏感内容。

### 8.4 上游真实缓存

只要 Kiro-Go 返回任意非零真实缓存读取或缓存创建用量，V2 必须保留上游原始
usage，不得重复分配。该规则不受账号、分组和价格变化影响。

## 9. Kiro-Go Docker 内网拓扑

### 9.1 目标拓扑

```text
Sub2API 容器
  -> http://kiro-go-pr131:8080
  -> Kiro-Go 容器

共享网络：sub2api_sub2api-network
```

Kiro-Go 保留自身 Compose 默认网络，同时加入外部网络
`sub2api_sub2api-network`。在共享网络上显式提供 DNS 别名
`kiro-go-pr131`，避免依赖 Compose 项目生成名称。

账号 `1910 / goep-kiro-power` 的 Base URL 改为：

```text
http://kiro-go-pr131:8080
```

这里必须使用 Kiro-Go 容器监听端口 `8080`，不能使用宿主机发布端口 `8321`。

### 9.2 宿主机端口

为保留服务器本机诊断能力，Kiro-Go 的端口发布改为：

```text
127.0.0.1:8321:8080
```

不得继续使用：

```text
0.0.0.0:8321:8080
```

如果确认没有任何宿主机本地工具依赖 `8321`，后续可以单独取消端口发布；本次
先收紧到回环地址，不扩大变更范围。

### 9.3 持久化方式

生产 `/home/ubuntu/kiro-go-pr131/docker-compose.yml` 必须写入外部网络、网络
别名和回环端口绑定。只执行一次临时 `docker network connect` 不能视为修复完成，
因为容器重建后会丢失。

应用 Compose 变更时只允许定向重建 Kiro-Go 服务。Sub2API、PostgreSQL、Redis、
Caddy、CPA、QQ/NapCat 和其他无关容器不得被重建。

## 11. 测试设计

### 11.1 配置测试

- 缺少 V2 账号开关时不启用。
- 账号开启后，不依赖分组 ID 判断资格。
- 账号开启后，缺少 `pricing_profile` 仍然启用。
- 旧 `pricing_profile` 保留时兼容运行。
- V1 旧开关仍不能启动 V2。

### 11.2 价格向量测试

- `$5/$25/$0.5/$6.25/$10` 能无损转换并执行分配。
- `$5/$25/$0.6/$6.25/$10` 能无损转换并执行分配。
- 切换分组只改变价格向量，不改变账号资格。
- 非正价格、缺失价格、区间价格和长上下文倍率触发单请求降级。
- 价格转换和 token 乘法溢出时安全降级。

### 11.3 分配器测试

- 读取型、单一小时创建型、混合创建型在 `$0.5` 和 `$0.6` 读取价格下都精确守恒。
- 同一请求和同一价格向量重试结果确定。
- 输出 token 永远保持不变。
- 上游真实缓存用量永远不被重新分配。
- 找不到受边界约束的精确整数解时退回原始用量。

### 11.4 响应与计费一致性测试

- 非流式和流式响应携带同一价格快照到 `ForwardResult`。
- 最终 `message_delta.usage` 使用动态价格分配。
- 响应完成后即使 API Key 分组改变，计费仍使用响应时价格快照。
- 响应完成后即使价格解析器刷新，计费结果仍不改变。
- 价格快照缺失、损坏或与分配结果不守恒时退回原始用量。
- 用户扣费、订阅消耗、API Key 配额和账号成本继续使用原始锁定费用。
- 普通使用日志显示对外用量，管理员审计字段保存原始用量。

### 11.5 Docker 内网测试

- Kiro-Go 同时连接默认网络和 `sub2api_sub2api-network`。
- Sub2API 容器可以解析 `kiro-go-pr131`。
- Sub2API 容器通过 `kiro-go-pr131:8080` 完成健康检查或真实请求。
- 账号 Base URL 不再包含生产公网 IP。
- 宿主机 `8321` 只监听 `127.0.0.1`。
- 公网不能直接访问生产 `8321`。

## 14. 回滚

代码、账号路由和 Docker 网络必须能够独立回滚。

### 14.1 V2 回滚

首选关闭账号开关：

```text
equivalent_cache_allocation_v2.enabled = false
```

关闭账号开关不需要重启 PostgreSQL、Redis、Caddy 或 Kiro-Go。

### 14.2 应用版本回滚

- 恢复上一份精确镜像摘要。
- 只重建 `sub2api`。
- 保留新增的可空审计字段和历史使用记录。

### 14.3 Kiro-Go 路由回滚

- 从备份恢复账号 Base URL。
- 从备份恢复 Kiro-Go Compose。
- 只定向重建 Kiro-Go。
- 验证旧地址可达后再移除共享网络。

如果内网请求失败但 Kiro-Go 容器健康，可先恢复账号 Base URL，避免把应用镜像回滚
和网络回滚捆绑在一起。
